# Design Critique — 측정으로 검수한다

디자인 검수는 취향 논쟁이 아니다. **측정 가능한 기준**으로 결함을 찾고, 재현 가능한 수정 경로를 제시한다. 핵심은 "존재 확인"이 아니라 **여러 산출물을 동시에 읽어 교차 비교**하는 것이다.

## 검수 4영역

### 1. 접근성 — 색 대비 (정량, 최우선)
모든 text/surface 색 조합을 `scripts/contrast.mjs`로 계산한다.

```bash
# 단건
node scripts/contrast.mjs "#71717a" "#ffffff"           # AA(4.5:1) 판정
node scripts/contrast.mjs "#71717a" "#ffffff" --large    # 큰 텍스트(3:1)

# 배치: pairs.json = [{ "fg":"#...", "bg":"#...", "large":false, "label":"text.muted on surface.base" }, ...]
node scripts/contrast.mjs --pairs pairs.json
```

토큰에서 모든 텍스트/배경 쌍을 뽑아 pairs.json을 만들고 배치 실행한다. 라이트·다크 **양쪽** 검사. AA 미달은 결함, 수정 명도 방향까지 제시.

### 2. 수학적 정합 (정량)
- 타이포 스케일이 선언 비율을 따르는가? `modular-scale.mjs`로 재생성해 대조하거나 인접 단계 비를 계산.
- 간격이 base unit의 배수인가?
- 라인하이트가 수직 리듬(baseline 그리드 배수)에 맞는가?

### 3. 토큰 무결성 (교차 비교)
- **tokens.json의 모든 `{별칭}`이 실재 토큰을 가리키는가** (끊긴 참조 탐지).
- **tokens.json ↔ tokens.css 값 일치** — 두 파일을 동시에 읽어 토큰별로 대조.
- **다크모드 누락** — 라이트 의미 토큰 집합과 다크 집합을 비교, 빠진 토큰 탐지.
- **가이드 ↔ 토큰 일치** — DESIGN_GUIDE.md가 인용한 값이 실제 토큰과 같은가.

### 4. 언어 일관성·트렌드 정렬 (정성)
- 산출물이 director가 고른 디자인 언어를 일관되게 따르는가 (예: 미니멀인데 화려한 그림자가 끼었는가).
- trend-researcher의 원리에서 벗어나지 않았는가.
- 정성 판단은 단정하지 않고 근거와 함께 director 판단에 위임.

## 증분 검증

전체 완성 후 1회가 아니라:
1. geometry 산출 직후 → 스케일·간격 정합 부분 검증
2. aesthetics 산출 직후 → 색 대비 부분 검증
3. 종합(tokens) 후 → 무결성·교차 비교 전체 검증

조기에 잡을수록 재종합 비용이 작다.

## 결함 보고 형식

`_workspace/04_critique.md`:

```markdown
## 검수 요약
- 통과 N / 실패 M (심각도별)

## 실패 항목
### [심각도: 치명/주의/경미] 항목명
- 증거: color.text.muted(#8a8a8a) on surface.base(#fff) = 3.45:1, AA 4.5:1 미달 (contrast.mjs)
- 영향: 본문 보조 텍스트 가독성
- 수정 경로: muted를 neutral.600(#52525b, L≈0.38)로 → 약 7:1
```

## 원칙

- **숫자가 먼저.** "어색하다"가 아니라 "비율 1.31로 선언값 1.25에서 벗어남". 스크립트 출력을 증거로 인용.
- **수정 경로 필수.** 결함마다 어떻게 고칠지 방향(명도·토큰 교체 등)을 제시.
- **재현 가능.** critic의 모든 정량 판정은 같은 입력에 같은 결과여야 한다.
